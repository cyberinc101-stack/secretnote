How It Works

Encrypted Notes — Your Browser Does the Encryption

Your note is encrypted before it leaves your device. The decryption key never touches our server. Here is exactly how it works.

Most note-sharing services encrypt data in transit (HTTPS) and at rest (disk encryption). That sounds secure, but it means the service can still read your notes — the server decrypts them whenever it needs to display them. SecretNote takes a different approach: your note is encrypted entirely in your browser, and the decryption key never reaches the server at all.

Server-side vs browser-side encryption

The distinction is critical:

  • Server-side encryption: The service encrypts your data using a key it controls. The server can decrypt your data any time. Employees, law enforcement requests, and attackers who compromise the server can all access your notes.
  • Browser-side (client-side) encryption: Your browser generates an encryption key, encrypts your data locally, and sends only ciphertext to the server. The server never sees the key and cannot decrypt the data — ever.

SecretNote uses browser-side encryption exclusively.

The encryption process, step by step

📝
Your browser generates a random 256-bit AES key using the Web Crypto API.
🔒
Your browser encrypts your note text with AES-256-GCM using that key. The result is unreadable ciphertext.
📤
Only the ciphertext is sent to the SecretNote server — the key stays in your browser.
🔗
The key is embedded in the URL fragment (#k=...) of the one-time link you receive.
🌐
When the recipient opens the link, their browser reads the key from the fragment — which is never sent in HTTP requests — and decrypts the note locally.
💥
The ciphertext is deleted from the server the moment the note is read. Nothing remains.

Why the URL fragment is the key insight

The URL fragment — everything after the # symbol — is a browser-only concept. When a browser requests a URL, the fragment is stripped before the HTTP request is sent. This is defined in the HTTP specification. It means:

  • Our server never receives the decryption key
  • Network monitoring tools (proxies, ISP logging) never see the key
  • Server access logs never record the key
  • No intermediary (CDN, load balancer, firewall) ever sees the key

The key exists only in your browser, the URL you copy, and the recipient's browser when they open the link. After the note is read and deleted, the key is gone everywhere.

AES-256-GCM: what makes it strong

AES-256-GCM (Galois/Counter Mode) is the encryption standard used by governments, banks, and security-critical applications worldwide. The "256" refers to the key length — 2²⁵⁶ possible keys, a number so large that brute-forcing it is computationally impossible with any hardware that could ever be built. The "GCM" mode adds authentication — any tampering with the encrypted data is detected before decryption, preventing certain classes of attack.

What the server actually stores

When you create a note, the server receives and stores:

  • A random note ID (to identify the blob)
  • The AES-256-GCM encrypted ciphertext (unreadable without the key)
  • An expiry timestamp (if you set one)
  • A flag for whether confirmation is required on open

That is all. No username, no IP linked to note content, no decryption key, no plaintext — not even a hash of the plaintext. The server is a temporary holding container for encrypted data it cannot read.

What this means for your privacy

  • SecretNote employees cannot read your notes — the infrastructure does not allow it
  • A data breach of our server exposes only unreadable ciphertext
  • A court order served on SecretNote could not produce your note content — we do not have it
  • There is no master key, no key escrow, and no key recovery — the key lives and dies with the URL

Frequently asked questions

Is the Web Crypto API trustworthy?

The Web Crypto API (window.crypto.subtle) is built into every modern browser and is the same cryptographic foundation used by password managers, secure messaging apps, and banking sites. It is not something SecretNote wrote — it is a standardised browser primitive audited and maintained by browser vendors.

Could SecretNote's JavaScript be compromised to send the key?

This is a valid theoretical concern with any client-side encryption system. If a malicious actor could replace the JavaScript served by SecretNote, they could in principle intercept keys before encryption. Mitigations include checking the open-source code, using browser extensions that block script changes, and for the highest-stakes uses, a self-hosted setup.

What happens if I lose the link?

The link is the only thing that contains the decryption key. If you lose it, the note is unrecoverable — even by us. Save the link immediately after creating a note if there is any chance you will need to verify it was received.
