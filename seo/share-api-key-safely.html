How to Share an API Key Safely

API keys and tokens pasted into Slack, email, or chat stay in those logs forever. Here is a better way.

Sharing API keys is one of the most common security mistakes in development teams. A key gets pasted into Slack to onboard a new hire, or emailed to a contractor, and it lives in that message thread indefinitely — searchable, accessible to admins, and exposed in any future breach of that platform.

⚠ The hidden danger of Slack and email Security audits frequently find API keys, tokens, and database passwords scattered across years of Slack history. A single compromised account gives an attacker access to all of it.

The secure way: a self-destructing encrypted link

SecretNote encrypts your secret in the browser using AES-256-GCM, stores only the ciphertext, and destroys it the moment the recipient opens the link. The decryption key lives only in the URL fragment — a part of the URL that is never sent to any server, including ours.

  1. Paste your API key or secret into SecretNote. Encryption happens locally in your browser before anything is transmitted.
  2. Click Create Note. You get a one-time link. The key in the URL fragment is invisible to servers and network monitors.
  3. Send the link to your colleague via Slack, email, or any channel. The link itself reveals nothing — only the encrypted blob ID.
  4. They click the link. The key is decrypted in their browser, displayed once, and the note is permanently deleted. If they try the link again, it is already gone.

Add a time limit for contractor handoffs

When sharing with a contractor or external developer, set an expiry. If they do not retrieve the key within 24 hours — or whatever window you choose — the note self-destructs automatically. There is no dangling secret waiting to be found later.

For the most sensitive secrets: double-layer protection

Use SecretNote's optional passphrase on top of the standard encryption. The flow looks like this:

  • Send the one-time link via Slack or email.
  • Send the passphrase separately via SMS or a quick call.
  • An attacker who intercepts either channel alone cannot access the secret.

This is especially useful for production credentials, billing API keys, or anything with financial exposure.

What to do after sharing

Even with a self-destructing link, follow these steps once the recipient has the key:

  • Rotate the key after the onboarding period if the service supports it — treat any shared secret as potentially exposed.
  • Scope the key to the minimum required permissions before sharing.
  • Store the key in a secrets manager (Vault, AWS Secrets Manager, Doppler) going forward rather than sharing it again by hand.

Frequently asked questions

Can SecretNote read my API key?

No. The AES-256-GCM encryption key exists only in the URL fragment, which is never sent to our server. We receive and store only ciphertext we cannot decrypt.

What if someone forwards the link?

A forwarded link is already dead once the first person opens it. Each link is strictly one-time-use. The second person to open it sees a "note not found" message.

Is there a size limit?

Up to 50,000 characters — more than enough for any key, token, certificate, or .env block.

Why not just use a password manager's sharing feature?

Password managers are excellent for long-term storage but typically require both parties to have accounts. SecretNote requires nothing — just a browser. Use it for one-off handoffs; use a password manager for ongoing shared credentials.
