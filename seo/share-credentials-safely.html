How to Share Login Credentials Safely

Emailing a password to someone is a habit that creates a permanent security liability. Here is what to do instead.

At some point everyone needs to hand off a set of login credentials — to a new team member, a freelancer, a family member managing accounts, or a client receiving access to a service you have set up for them. The method you choose matters enormously for your long-term security.

What not to do

✗ EmailSits in the sent folder, inbox, and server backups indefinitely. Any future account breach exposes it.
✗ Slack / Teams messageSearchable by admins, retained in company logs, exposed in any platform breach.
✗ WhatsApp / iMessageBacked up to iCloud or Google Drive unless manually disabled. Visible on all synced devices.
✗ Plain text documentEasily forwarded, copied, or indexed by backup systems.

What to do instead

✓ Self-destructing encrypted noteThe credential is encrypted and permanently deleted the moment the recipient reads it. Nothing remains.
✓ Password manager shareServices like 1Password and Bitwarden have built-in secure sharing. Requires both parties to have accounts.

For quick handoffs with external people who do not share your password manager, a self-destructing encrypted note is the most practical option — it requires nothing from the recipient except a browser.

How to share credentials using SecretNote

  1. Open SecretNote and type the credentials in a readable format — username, password, URL, and any notes the recipient needs. Everything is encrypted in your browser before it is stored.
  2. Optionally set an expiry window. For onboarding, 24 hours or 7 days is practical. For emergency access handoffs, 1 hour is appropriate.
  3. For extra security, add an optional passphrase. The note requires this password to decrypt.
  4. Click Create Note to get a one-time link. Send the link over email, Slack, or any channel — the link itself contains only an encrypted blob ID.
  5. If you added a passphrase, send it separately — via SMS, a call, or in person.
  6. The recipient opens the link, enters the passphrase if required, reads the credentials once, and the note is permanently deleted. The link is dead from that moment on.
Pro tip: split the secret across channels Send the encrypted link over email and the passphrase over text. An attacker who intercepts either channel alone gets nothing. This is especially valuable for shared accounts with financial or administrative access.

Credential types this works for

  • Website logins (username + password)
  • Admin accounts and control panel access
  • Database credentials
  • Wi-Fi passwords
  • VPN credentials and configuration
  • Social media or advertising account access
  • Email account setup credentials
  • Software license keys and activation codes
  • Shared team accounts (billing, analytics, hosting)

After the handoff: what to do next

A self-destructing note eliminates the trail, but good credential hygiene continues after the handoff:

  • Rotate the password after onboarding is complete, especially if it was a shared credential.
  • Use individual accounts where possible — sharing one login between multiple people creates accountability gaps.
  • Move to a password manager for ongoing credential management, rather than sharing the same password again by hand in the future.
  • Enable two-factor authentication on any account with significant access, independent of the password handoff method.

Hand off those credentials the safe way — encrypted, self-destructing, and gone the moment they are read.

Frequently asked questions

Can SecretNote read the credentials I share?

No. Your note is encrypted in your browser using AES-256-GCM before it is stored. The decryption key lives only in the URL fragment, which browsers never send to our server. We cannot decrypt anything you create.

What if the recipient accidentally closes the tab before saving the password?

Once a note is read and destroyed, it cannot be retrieved. Remind recipients to save the credentials to a password manager before closing the tab. For sensitive credentials, you might share them to a temporary secondary email address the recipient can access from multiple devices.

Is this compliant with company security policies?

SecretNote's browser-side encryption means credentials are never exposed on an external server in readable form — which addresses most security policy concerns about third-party sharing. For environments with strict data governance requirements, check with your security team. Self-hosted alternatives are available for teams with specific compliance needs.
